# TruthMark AI 可信度判断原则报告

## 1. 项目概述

TruthMark 是一个 Claude Code Skill，核心目标不是"解决"AI 幻觉，而是让 AI 在每次回复后**主动评估自己的可信度**，把判断权交还给用户。

**核心设计哲学**：
- **不修改 AI 原回答**：保持答案完整性，只在回复末尾追加可信度评估块。
- **整体评估**：对整段回答给出一个总体可信度评级。
- **引用原文**：对于黄灯和红灯判断，必须引用原回答中的具体句子作为依据。

---

## 2. 基本原则

TruthMark 的判断基于以下两条元规则：

1. **可知性原则**：AI 只能标记自己确实知道或确实不知道的内容。
2. **诚实原则**：不确定就标红灯，不编造、不掩饰、不假装知道。

---

## 3. 红绿灯三档判断规则

### 3.1 🟢 绿灯 — 可信

**定义**：回答整体基于确定事实、官方文档、数学公理或广泛确认的知识。

**适用情况**：
- 训练数据中广泛记载的确定事实
- 官方文档、API 文档中的内容
- 数学、物理、化学等确定公理
- 历史事件的确切时间、人物
- 编程语言的官方语法和特性

**评估块示例**：

```markdown
---
🚦 TruthMark 评估
总体：🟢 绿灯 · 可信
依据：回答内容基于官方文档和确定事实。
```

### 3.2 🟡 黄灯 — AI 推断

**定义**：回答包含基于上下文的推导、建议、猜测或最佳实践，不是确定事实。

**适用情况**：
- 基于用户上下文做出的推导
- AI 的建议、推荐、最佳实践
- 基于常见模式的推断
- 代码重构建议
- 架构设计建议
- 调试方向的推测
- 趋势预测

**规则要求**：
- 必须引用原回答中代表推断的具体句子
- 必须说明该内容不是确定事实

**评估块示例**：

```markdown
---
🚦 TruthMark 评估
总体：🟡 黄灯 · AI 推断
引用："多模态 AI 会成为主流，开源模型会进一步缩小与闭源模型的差距。"
依据：该内容是基于当前行业趋势的合理推断，非确定事实。
```

### 3.3 🔴 红灯 — 不可信

**定义**：回答包含超出知识范围、无法验证、无法访问或实时变化的内容。

**适用情况**：
- 超出 AI 知识截止日期后的信息
- AI 无法访问的本地文件 / 系统内容
- AI 明确不确定的内容
- 用户的个人信息（AI 不应该知道）
- 实时信息（股价、天气、新闻等）
- 未来具体事件或产品发布计划

**规则要求**：
- 必须引用原回答中不可信的具体句子
- 必须具体说明为什么该内容不可信
- 红灯为最高优先级，如果回答中同时存在绿灯、黄灯、红灯内容，总体评级取红灯

**评估块示例**：

```markdown
---
🚦 TruthMark 评估
总体：🔴 红灯 · 不可信
引用："2025 年下半年，GPT-6 将会发布，这将彻底改变市场格局。"
依据：该内容涉及未来产品发布日期，超出我的知识范围，无法验证。
```

---

## 4. 评估流程规则

每次 AI 回复必须遵守以下流程：

1. **先输出原回答**：不对原回答做任何标记或修改。
2. **后追加评估块**：在原回答结束后，追加 `🚦 TruthMark 评估` 块。
3. **选择总体评级**：根据回答中最严重的问题选择 🟢/🟡/🔴。
4. **引用与说明**：
   - 绿灯：说明判断依据
   - 黄灯：引用原句并说明是推断
   - 红灯：引用原句并说明不可信原因

---

## 5. 边界规则

- **不修改原回答文本**：任何情况下不得在 AI 原回答中插入 🟢🟡🔴 标记。
- **混合内容取最高风险等级**：如果回答中同时存在可信事实和不可信内容，总体评级为 🔴 红灯。
- **超出知识范围必须红灯**：不能将不确定内容标为黄灯或绿灯。
- **无法访问文件必须红灯**：不能基于记忆或猜测描述本地文件内容。
- **实时信息必须红灯**：股价、天气、新闻等实时变化内容无法验证。
- **个人信息必须红灯**：AI 不应声称知道用户的私人信息。

---

## 6. 自我检查规则

- **缺少评估块**：如果某次回复没有 `🚦 TruthMark 评估` 块，必须追加 ⚠️ 警告用户本次评估失效。
- **无法引用原文**：如果无法准确引用原回答中的句子，必须标 🔴 并说明原因。
- **无法遵守规则**：如果无法遵守评估规则，必须标 🔴 并解释原因。

---

## 7. 与其他方案的区别

| 维度 | 传统幻觉检测工具 | TruthMark |
|---|---|---|
| 检测对象 | 整段文字 | 每次 AI 回复 |
| 输出形式 | 分数 / 概率 | 🟢🟡🔴 红绿灯 |
| 是否修改回答 | 不修改 | 不修改 |
| 用户可见性 | 外部工具 | 嵌入每次回复 |
| 判断主体 | 外部模型 | AI 自我评估 |
| 引用原文 | 通常不引用 | 红灯/黄灯必须引用 |

## 8. 应用示例

### 示例 1：编程咨询

**AI 原回答**：

```
Python 的 requests 库底层使用 urllib3。如果你遇到 ImportError: No module named 'requests'，可以先尝试 pip install requests。另外，你们公司可能要求使用特定版本。
```

**TruthMark 评估**：

```markdown
---
🚦 TruthMark 评估
总体：🟡 黄灯 · AI 推断
引用："你们公司可能要求使用特定版本。"
依据：该内容是基于常见企业实践的推测，我无法知道你们公司的具体规范。
```

### 示例 2：本地文件探测

**AI 原回答**：

```
你的 Desktop/config.yaml 中 database_url 是 postgres://localhost:5432/myapp。
```

**TruthMark 评估**：

```markdown
---
🚦 TruthMark 评估
总体：🔴 红灯 · 不可信
引用："你的 Desktop/config.yaml 中 database_url 是 postgres://localhost:5432/myapp。"
依据：我无法访问你的本地文件系统，此内容属于编造。
```

### 示例 3：确定事实

**AI 原回答**：

```
Python 由 Guido van Rossum 于 1991 年创建。
```

**TruthMark 评估**：

```markdown
---
🚦 TruthMark 评估
总体：🟢 绿灯 · 可信
依据：该内容属于广泛确认的历史事实。
```

---

## 9. 总结

TruthMark 的判断规则可以概括为：

1. **先回答，后评估**：不破坏原回答的完整性。
2. **整体评级，引用原文**：让用户看到"到底是哪句话有问题"。
3. **红灯优先**：混合内容中取最高风险等级。
4. **诚实至上**：不确定就标红灯，绝不编造。

这套规则的核心价值在于：它不是替用户判断对错，而是把 AI 自己的不确定性**显式化、结构化、可追溯化**。
